# Design Handoff Checklist

**Purpose:** Validate clean handoffs between Design Squad agents
**Agent:** Design Chief (Orchestrator)
**Standard:** Complete context transfer with zero information loss

---

## SECTION 1: PROJECT CONTEXT TRANSFER

Ensure receiving agent has full understanding of project background.

### 1.1 Original Request
- [ ] Original user request documented verbatim
- [ ] Primary objective clearly stated
- [ ] Success criteria defined and measurable
- [ ] Any user preferences or constraints noted

**Pass Criteria:** Receiving agent can restate the project goal accurately.

### 1.2 Project History
- [ ] Previous phases documented (if multi-phase project)
- [ ] Decisions made in earlier phases captured
- [ ] Rationale for decisions included
- [ ] Any pivots or scope changes noted

**Pass Criteria:** Receiving agent understands why project is at current state.

### 1.3 Stakeholder Context
- [ ] End user/audience identified
- [ ] Business context provided
- [ ] Approval chain documented (if applicable)
- [ ] Communication preferences noted

**Pass Criteria:** Receiving agent knows who they're ultimately serving.

---

## SECTION 2: DELIVERABLES HANDOFF

Verify all work products are properly transferred.

### 2.1 File Inventory
- [ ] All files listed with clear naming
- [ ] File locations/paths specified
- [ ] File formats appropriate for next phase
- [ ] Version numbers indicated (if applicable)

**Pass Criteria:** Receiving agent can locate all files immediately.

### 2.2 Deliverable Status
- [ ] Each deliverable marked: COMPLETE | IN_PROGRESS | BLOCKED
- [ ] Quality gate status for each: PASSED | PENDING | FAILED
- [ ] Any open issues flagged
- [ ] Review status documented

**Pass Criteria:** Receiving agent knows exactly what's ready vs pending.

### 2.3 Source Files
- [ ] Editable source files included (not just exports)
- [ ] Font files or references provided
- [ ] Asset libraries linked
- [ ] Design tool access confirmed (Figma, etc.)

**Pass Criteria:** Receiving agent can modify work if needed.

---

## SECTION 3: CONSTRAINTS AND REQUIREMENTS

Transfer all limitations and specifications.

### 3.1 Technical Constraints
- [ ] Platform/medium requirements documented
- [ ] Size/resolution specifications noted
- [ ] Format requirements specified
- [ ] Performance considerations flagged

**Pass Criteria:** Receiving agent won't violate technical limits.

### 3.2 Brand Constraints
- [ ] Brand guidelines referenced or attached
- [ ] Color palette documented (with hex/RGB values)
- [ ] Typography rules specified
- [ ] Voice and tone guidelines included

**Pass Criteria:** Receiving agent will maintain brand consistency.

### 3.3 Timeline Constraints
- [ ] Deadline for next phase specified
- [ ] Milestones identified
- [ ] Dependencies on other work noted
- [ ] Buffer time (if any) communicated

**Pass Criteria:** Receiving agent understands time expectations.

### 3.4 Budget Constraints (If Applicable)
- [ ] Budget scope for next phase defined
- [ ] Already spent vs remaining documented
- [ ] Cost limitations noted
- [ ] Approval thresholds specified

**Pass Criteria:** Receiving agent won't exceed budget.

---

## SECTION 4: DECISIONS AND RATIONALE

Document the "why" behind the "what".

### 4.1 Key Decisions
- [ ] Major design decisions listed
- [ ] Alternatives considered documented
- [ ] Selection rationale explained
- [ ] User feedback incorporated (if any)

**Pass Criteria:** Receiving agent understands why choices were made.

### 4.2 Trade-offs Made
- [ ] Compromises documented
- [ ] What was prioritized vs deprioritized
- [ ] Risks accepted noted
- [ ] Areas left for future iteration flagged

**Pass Criteria:** Receiving agent won't redo resolved trade-offs.

### 4.3 Open Questions
- [ ] Unresolved questions listed
- [ ] Who needs to answer them identified
- [ ] Impact of delayed answers noted
- [ ] Assumptions made in absence of answers documented

**Pass Criteria:** Receiving agent knows what's still uncertain.

## SECTION 6: COMMUNICATION PROTOCOL

Establish clear communication for handoff period.

### 6.1 Handoff Meeting (If Needed)
- [ ] Synchronous handoff scheduled (for complex projects)
- [ ] Key points to cover listed
- [ ] Questions from receiving agent collected
- [ ] Demo/walkthrough planned

**Pass Criteria:** Both agents aligned after handoff meeting.

### 6.2 Async Handoff Document
- [ ] Handoff summary document created
- [ ] All sections of this checklist addressed
- [ ] Document location shared
- [ ] Confirmation of receipt obtained

**Pass Criteria:** Receiving agent confirms understanding.

### 6.3 Ongoing Communication
- [ ] Clarification channel established
- [ ] Response time expectations set
- [ ] Escalation path defined
- [ ] Check-in schedule agreed (if multi-week)

**Pass Criteria:** Both agents know how to reach each other.

---

## SECTION 7: QUALITY GATES

Verify handoff meets quality standards.

### 7.1 Completeness Check
- [ ] No placeholder content in deliverables
- [ ] All promised deliverables included
- [ ] Documentation complete (not "TBD")
- [ ] No broken links or missing files

**Pass Criteria:** Handoff is complete, not partial.

### 7.2 Accuracy Check
- [ ] File names match inventory
- [ ] Versions are current (not outdated)
- [ ] Information is consistent across documents
- [ ] No contradictions between sections

**Pass Criteria:** Handoff information is accurate.

### 7.3 Usability Check
- [ ] Receiving agent can open all files
- [ ] Tools/access available to receiving agent
- [ ] Information organized logically
- [ ] Easy to find specific information

**Pass Criteria:** Receiving agent can work efficiently.

## HANDOFF TEMPLATE

Use this template for documenting handoffs:

```yaml
handoff_document:
  metadata:
    project: "{Project Name}"
    date: "{YYYY-MM-DD}"
    from_agent: "@{sending_agent}"
    to_agent: "@{receiving_agent}"
    phase_completed: "{Phase name}"
    phase_starting: "{Phase name}"

  project_context:
    original_request: |
      {Verbatim user request}
    objective: "{Primary objective}"
    success_criteria:
      - "{Criterion 1}"
      - "{Criterion 2}"
    constraints:
      timeline: "{Deadline if any}"
      technical: "{Technical constraints}"
      brand: "{Brand constraints}"

  deliverables:
    files:
      - name: "{Filename}"
        path: "{Location}"
        format: "{Format}"
        status: "COMPLETE"
    pending:
      - "{Any incomplete items}"

  decisions_made:
    - decision: "{What was decided}"
      rationale: "{Why}"
      alternatives: "{What was considered}"

  open_questions:
    - question: "{Unresolved question}"
      impact: "{Impact on next phase}"
      owner: "{Who should answer}"

  specialist_notes:
    domain_context: |
      {Domain-specific information}
    references:
      - "{Reference 1}"
    warnings:
      - "{Things to watch out for}"

  communication:
    clarification_channel: "{How to reach sending agent}"
    response_time: "{Expected response time}"
    next_checkpoint: "{Date/milestone for check-in}"

  signatures:
    sending_agent:
      confirmed: true
      date: "{Date}"
    receiving_agent:
      confirmed: true
      date: "{Date}"
    design_chief:
      validated: true
      date: "{Date}"
```

---

## QUICK REFERENCE: MINIMUM VIABLE HANDOFF

For quick/simple handoffs, ensure at minimum:

1. [ ] Original request documented
2. [ ] Success criteria defined
3. [ ] All files located and accessible
4. [ ] Key constraints noted
5. [ ] Receiving agent confirms understanding

---

## NOTES

- Handoff quality directly impacts next phase success
- Incomplete handoffs cause delays and rework
- When in doubt, over-document rather than under-document
- Design Chief validates all handoffs before proceeding
- Store handoff documents in project folder for reference
